# Исследование: индексированный поиск по коду для VS Code

Дата среза: 2026-07-21.

## Цель

Проверить, есть ли продуктовая и техническая ниша для расширения VS Code с пользовательским опытом уровня PhpStorm: единый поиск файлов, символов, текста и действий, постоянный локальный индекс, мгновенная выдача и полноценная навигация с клавиатуры.

## Бизнес-ценность

Основная ценность — экономия времени разработчика на повторяющихся поисковых действиях в больших рабочих областях. Потенциальное преимущество формируется не отдельной функцией, а сочетанием четырёх качеств:

1. постоянный индекс содержимого;
2. единое окно поиска;
3. качественное ранжирование результатов разных типов;
4. надёжная работа индекса без регулярных полных пересборок.

Рынок пока выглядит фрагментированным: отдельные расширения умеют хороший интерфейс без индекса, индекс без зрелого интерфейса или поиск файлов и символов без полнотекстового индекса. Это возможность, но не доказательство большого спроса.

## Что делает PhpStorm

### Факты

- `Search Everywhere` открывается двойным нажатием Shift и объединяет файлы, классы, символы, действия, настройки и Git в одном входе. Категории переключаются клавишей Tab. Есть отдельные быстрые команды для классов, файлов, символов и действий. Источник: [JetBrains: Search Everywhere](https://www.jetbrains.com/help/phpstorm/searching-everywhere.html).
- Полнотекстовый поиск поддерживает регистр, целые слова, регулярные выражения, маски файлов и области поиска. Источник: [JetBrains: Find and replace text in project](https://www.jetbrains.com/help/phpstorm/finding-and-replacing-text-in-project.html).
- Анализ проекта строит карту классов, методов, объектов и других элементов кода. Она обслуживает навигацию, завершение, инспекции, переработку кода, поиск использований и другие интеллектуальные функции. Источник: [JetBrains: Project analysis](https://www.jetbrains.com/help/phpstorm/project-analysis.html).
- Платформа IntelliJ разделяет файловые индексы содержимого и индексы компактных заглушек объявлений. Во время построения индекса часть интеллектуальных функций ограничена. Источник: [IntelliJ Platform: Indexing and PSI Stubs](https://plugins.jetbrains.com/docs/intellij/indexing-and-psi-stubs.html).
- PhpStorm предоставляет исключения, именованные области, общие заранее построенные индексы и сценарии восстановления повреждённого индекса. Источники: [Excluded files](https://www.jetbrains.com/help/phpstorm/excluding-files-from-project.html), [Scopes](https://www.jetbrains.com/help/phpstorm/configuring-scopes-and-file-colors.html), [Shared indexes](https://www.jetbrains.com/help/phpstorm/shared-indexes.html), [Repair IDE](https://www.jetbrains.com/help/phpstorm/repair-ide.html).

### Вывод

Копировать только окно `Search Everywhere` недостаточно. Ощущение «поиска как в PhpStorm» создают индекс, области, ранжирование, предпросмотр, переходы с клавиатуры, прозрачное состояние анализа и восстановление после сбоев.

## Сигналы пользователей

Это не строгая статистика рынка, а повторяющиеся симптомы из открытых обсуждений.

- Пользователи, перешедшие с PhpStorm, прямо спрашивают об индексированном аналоге поиска VS Code для больших проектов: [Stack Overflow](https://stackoverflow.com/questions/40611840/how-can-i-make-vs-codes-search-in-files-go-faster/76957600).
- При удалённой работе первый поиск по тексту и файлам может быть заметно медленным; пользователь просит предварительное индексирование: [Stack Overflow](https://stackoverflow.com/questions/78075380/how-to-tell-vscode-to-index-the-entire-project-folder).
- Поставщики языковых функций могут иметь собственные проблемы с производительностью поиска символов, поэтому единый интерфейс не должен считать LSP всегда быстрым: [Pyright #1997](https://github.com/microsoft/pyright/issues/1997).
- Для больших рабочих областей C/C++ встречаются жалобы на повторное сканирование после запуска: [vscode-cpptools #13927](https://github.com/microsoft/vscode-cpptools/issues/13927).
- Пользователи ищут аналог `Search Everywhere`, потому что `Ctrl+T` покрывает только символы: [обсуждение Reddit](https://www.reddit.com/r/vscode/comments/hab1k1).
- У PhpStorm тоже есть болезненный сценарий: циклическая или дорогая повторная индексация. Это предупреждение для нашего жизненного цикла индекса, а не преимущество, которое следует повторять: [Stack Overflow](https://stackoverflow.com/questions/78400438/continuous-indexing-issue-in-phpstorm).

## Возможности и ограничения VS Code

### Доступно в стабильном API

- Поиск файлов, наблюдатели файлов, события изменений открытых документов, частное хранилище рабочей области и собственный интерфейс `Quick Pick` или `Webview`: [VS Code API](https://code.visualstudio.com/api/references/vscode-api).
- Запрос ко всем поставщикам символов рабочей области через `vscode.executeWorkspaceSymbolProvider`: [Built-in Commands](https://code.visualstudio.com/api/references/commands).
- Расширение типа `workspace` выполняется рядом с файлами в WSL, SSH и контейнерах. Это правильное место для индекса: [Remote Development and Codespaces](https://code.visualstudio.com/api/advanced-topics/remote-extensions).
- Расширение может работать в ограниченном режиме доверия, если только читает файлы и не запускает код проекта: [Workspace Trust](https://code.visualstudio.com/api/extension-guides/workspace-trust).

### Ограничения

- Программный полнотекстовый поиск `workspace.findTextInFiles` и поставщик текстового поиска всё ещё относятся к предлагаемому API. Расширение Marketplace не может строить основную функцию на таком API: [proposed findTextInFiles](https://github.com/microsoft/vscode/blob/main/src/vscode-dts/vscode.proposed.findTextInFiles.d.ts), [Using Proposed API](https://code.visualstudio.com/api/advanced-topics/using-proposed-api).
- Расширение не может внедряться в DOM VS Code. Для интерфейса сложнее стандартного списка нужен `Webview`: [Extension Capabilities](https://code.visualstudio.com/api/extension-capabilities/overview).
- Рекурсивные наблюдатели могут сворачивать события, а удаление каталога не обязано давать событие для каждого вложенного файла. Нужна периодическая сверка состояния, а не только события: [FileSystemWatcher](https://code.visualstudio.com/api/references/vscode-api).
- Нативные модули усложняют поддержку Windows, macOS, Linux, архитектур процессора, WSL, SSH, контейнеров и Alpine. Источник: [Extension Host](https://code.visualstudio.com/api/advanced-topics/extension-host).
- Полноценная браузерная версия расширения не имеет Node API и прямого доступа к диску. Её разумно не включать в первую версию: [Web Extensions](https://code.visualstudio.com/api/extension-guides/web-extensions).

### Продуктовая ниша

Ниша уже не формулируется как «первый индексированный поиск для VS Code». Защищаемая формулировка:

> Первый цельный локальный поиск для VS Code, который объединяет постоянный полнотекстовый индекс, файлы, символы и действия с пользовательским опытом уровня PhpStorm и надёжной работой в больших и удалённых рабочих областях.

Это гипотеза. Её нужно проверить прототипом и интервью, а не считать доказанной по количеству обсуждений.

## Архитектурные аналоги

- `ripgrep` очень быстро сканирует файлы и уважает правила исключений, но выполняет сканирование для каждого запроса. Он подходит как резервный механизм и средство точной проверки: [ripgrep](https://github.com/BurntSushi/ripgrep).
- `SQLite FTS5` имеет триграммный токенизатор для поиска подстрок через `MATCH`, `LIKE` и `GLOB`. Короткие запросы и некоторые шаблоны переходят к сканированию: [SQLite FTS5](https://www.sqlite.org/fts5.html).
- `Zoekt` — специализированный поиск по исходному коду с позиционными триграммами, шардированным индексом, поиском подстрок и регулярных выражений. Его индекс обычно заметно больше исходного текста: [Zoekt](https://github.com/sourcegraph/zoekt), [описание формата и ранжирования](https://github.com/sourcegraph/zoekt/blob/main/doc/design.md).
- GitHub пришёл к отдельному движку поиска по коду после ограничений общих поисковых систем. Это подтверждает, что кодовый поиск имеет особые требования к точным строкам и регулярным выражениям: [The technology behind GitHub’s new code search](https://github.blog/engineering/the-technology-behind-githubs-new-code-search/).

## Предлагаемое направление

### Компоненты

1. `Index Coordinator` перечисляет файлы, применяет исключения, строит первичный индекс, возобновляет прерванную работу и проводит периодическую сверку.
2. `Content Index` хранит имена файлов, метаданные, триграммы или FTS-записи и версию формата.
3. `Document Overlay` содержит актуальный текст несохранённых открытых документов поверх дискового индекса.
4. `Query Planner` разбирает режим, область, маски, регистр, целые слова и регулярное выражение; выбирает индекс или резервный поиск.
5. `Verifier` проверяет кандидатов по актуальному тексту и вычисляет точные строки и диапазоны.
6. `Symbol Adapter` получает символы через стабильных поставщиков языка VS Code.
7. `Ranker` объединяет совпадения по имени файла, границам слова, типу результата, недавней активности и качеству совпадения.
8. `Search UI` показывает группы, состояние индекса, предпросмотр и поддерживает полную клавиатурную навигацию.

### Техническая проверка до выбора движка

Сравнить на одинаковых наборах данных:

- `SQLite FTS5` с триграммным токенизатором через WebAssembly;
- специализированный шардированный триграммный индекс с точной проверкой.

Измерить:

- время холодной и повторной индексации;
- задержку первого результата и полной первой страницы;
- запросы длиной 1–2 символа, подстроки, Unicode, регистр, целые слова и регулярные выражения;
- размер индекса;
- пиковую память;
- блокировки процесса расширений;
- точечное обновление, переименование и удаление каталога;
- восстановление после принудительного завершения;
- Windows, macOS, Linux, WSL, SSH и контейнер.

## Объём первой версии

### Входит

- постоянный локальный индекс файлов и текстового содержимого;
- поиск подстроки, регистр и целое слово;
- маски включения и исключения;
- области: вся рабочая область, текущий каталог, открытые файлы;
- файлы, текст, символы и действия в едином окне;
- предпросмотр, группировка, открытие рядом и управление с клавиатуры;
- частичная выдача во время индексации и резервный поиск;
- автоматические точечные обновления;
- состояние, пауза, пересборка, проверка и восстановление индекса;
- несколько корней рабочей области;
- локальная работа без телеметрии по умолчанию.

### Не входит

- семантический поиск и векторные представления;
- граф вызовов, поиск использований и самостоятельный анализ языков;
- массовая замена;
- общие индексы между компьютерами;
- индекс истории Git и нескольких веток;
- браузерная версия расширения;
- облачная синхронизация;
- собственный сервер языковых функций.

## Предварительные критерии

Это целевые пороги для испытаний, а не подтверждённые характеристики:

- первая страница обычного запроса длиной от трёх символов: `p95 ≤ 100 мс` после прогрева;
- обновлённый сохранённый файл появляется в выдаче: `p95 ≤ 1 с`;
- повторный запуск не вызывает полную пересборку без изменения формата или повреждения;
- индекс занимает не более `3×` объёма индексируемого текста;
- поиск работает во время первичной индексации;
- повреждение или несовместимая версия обнаруживаются автоматически и приводят к безопасной пересборке;
- исходный код и запросы не покидают машину;
- операции индексации не должны надолго блокировать процесс расширений.

## Риски

- `FTS5` может дать быстрый прототип, но ограничить регулярные выражения, ранжирование и короткие подстроки.
- Собственный формат даст контроль, но резко увеличит объём реализации и тестирования.
- WebAssembly упрощает упаковку, но может уступать нативному движку по скорости и конкурентному доступу.
- Поставщики символов принадлежат другим расширениям и имеют непредсказуемую скорость и полноту.
- Наблюдатели файлов не являются достаточным источником истины.
- Огромный индекс способен повторить болезненный опыт PhpStorm, если нет паузы, прозрачного прогресса, ограничений ресурсов и восстановления.
- Количество установок существующих решений невелико; спрос нужно проверять отдельно.

2. Нужен ли в первой версии режим регулярных выражений или его следует добавить после устойчивого буквального поиска.
3. Какой набор открытых репозиториев использовать как постоянный эталон производительности.
4. Как распределить работу между процессом расширений и фоновым рабочим потоком без ухудшения интерфейса.
5. Считать ли удалённые рабочие области обязательным условием первого выпуска или критерием до публичной версии.
6. Нужна ли совместимость с Cursor и другими редакторами на основе VS Code уже в первой версии.
