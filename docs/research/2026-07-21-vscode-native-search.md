# Техническая граница использования родного поиска VS Code

Дата среза: 2026-07-21.

## Цель

Определить, какие части пользовательского сценария можно построить поверх VS Code без собственного поискового движка и без непубликуемого предлагаемого API.

## Матрица возможностей

| Возможность | Состояние | Применение |
|---|---|---|
| `workspace.findFiles` | стабильный API | Допустим для вспомогательных проверок, но основную выдачу файлов лучше оставить `Quick Open` |
| `vscode.executeWorkspaceSymbolProvider` | стабильная встроенная команда | Допустима для отдельных сценариев символов |
| `window.createQuickPick` | стабильный API | Допустим для небольшого диспетчера режимов |
| `workbench.action.quickOpen` | встроенная команда рабочего интерфейса | Основной вход в файлы и префиксы режимов; требуется проверка совместимости |
| `workbench.action.quickTextSearch` | родная обычная команда с VS Code 1.89 | Основной быстрый текстовый поиск |
| `workbench.action.findInFiles` | встроенная команда рабочего интерфейса | Расширенный текстовый поиск с параметрами |
| `workspace.findTextInFiles` | предлагаемый API | Не использовать в опубликованной версии |
| `search.action.getSearchResults` | внутренняя команда, возвращающая форматированный текст | Не использовать как контракт |
| Доступ к DOM родного окна | не поддерживается | Не использовать |

## Родные префиксы

`workbench.action.quickOpen` принимает начальную строку. Это позволяет открыть:

- файлы: `query`;
- символы рабочей области: `#query`;
- текст рабочей области: `%query`;
- действия: `>query`;
- справку по режимам: `?`.

Файлы и символы могут появляться вместе при включённой настройке `search.quickOpen.includeSymbols`.

## Расширенный текстовый поиск

Текущий исходный код `workbench.action.findInFiles` объявляет параметры:

- `query`;
- `replace`;
- `preserveCase`;
- `triggerSearch`;
- `filesToInclude`;
- `filesToExclude`;
- `isRegex`;
- `isCaseSensitive`;
- `matchWholeWord`;
- `useExcludeSettingsAndIgnoreFiles`;
- `onlyOpenEditors`;
- `showIncludesExcludes`.

Расширение должно передавать только эти параметры и не пытаться самостоятельно трактовать запрос.

## Почему нельзя собрать единый текстовый список

`Quick Search` использует внутреннюю поисковую службу VS Code, но публичный стабильный API не отдаёт его совпадения расширению. `workspace.findTextInFiles` находится в `vscode.proposed.findTextInFiles.d.ts`, а задача стабилизации остаётся открытой.

Варианты, которые отвергаются:

1. Разбирать результат `search.action.getSearchResults` — это неструктурированный внутренний формат без гарантии завершения поиска.
2. Читать DOM родного окна — расширения не должны внедряться во внутренний интерфейс VS Code.
3. Запускать `ripgrep` самостоятельно — это уже отдельный поисковый слой.
4. Читать все файлы через `workspace.fs` — это дублирование движка и нарушение границы продукта.
5. Использовать предлагаемый API — опубликованное расширение не должно от него зависеть.

## Рекомендуемая форма первой версии

### Команды пользователя

- «Открыть поиск»;
- «Найти файл или символ»;
- «Найти текст»;
- «Найти действие»;
- «Открыть расширенный поиск»;
- «Повторить последний режим».

### Состояние расширения

Допустимо хранить:

- последний выбранный режим;
- включённость заполнения из выделения;
- задержку двойного Shift;
- сохранённые наборы параметров расширенного поиска;
- сведения о последней успешной проверке совместимости.

Запрещено хранить:

- содержимое файлов;
- списки текстовых совпадений как постоянную базу;
- индекс файлов или символов;
- поисковые запросы в телеметрии;
- данные проекта за пределами локального состояния VS Code.

## Слой совместимости

Все вызовы `workbench.*` должны быть собраны в одном небольшом модуле. Он обязан:

1. Проверять наличие команды через `commands.getCommands`.
2. Передавать параметры без преобразования смысла.
3. Перехватывать ошибку вызова и предлагать соответствующую родную команду.
4. Иметь проверку для минимальной и текущей версии VS Code.
5. Не обращаться к внутренним объектам поиска и DOM.

## План технического опыта

1. Открыть каждый префикс через `workbench.action.quickOpen` с заранее известной строкой.
2. Проверить файл, символ, текст и действие на одной рабочей области.
3. Передать полный набор параметров в `workbench.action.findInFiles` и сверить поля панели.
4. Проверить выделенный текст и слово под курсором.
5. Проверить двойной Shift и конфликт с другим сочетанием.
6. Проверить открытие результата, открытие рядом, отмену и переход из `Quick Search` в `Search`.
7. Повторить на Windows и в одной удалённой среде.
8. Повторить на минимальной и текущей поддерживаемой версии VS Code.

## Условие перехода к реализации

Реализация начинается, если технический опыт подтверждает:

- четыре родных режима открываются с переданным запросом;
- расширенный поиск получает параметры без потери;
- двойной Shift имеет безопасный запасной вызов;
- нет необходимости читать файлы или результаты поиска;
- путь пользователя короче и понятнее набора отдельных стандартных сочетаний.
